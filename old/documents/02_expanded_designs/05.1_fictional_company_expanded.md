# 05.1 The Evolving Fictional Company (Expanded Design)

## Core Concept
A multi-modal simulation of a tech company's lifecycle. Rather than a single linear narrative, this system generates a web of interconnected corporate artifacts—Slack logs, JIRA tickets, Git commits, and angry executive emails.

## Detailed Mechanics

### 1. Architecture as State
The central source of truth is not a prose summary, but a mock system architecture and organization chart.
- **`org_chart.json`**: Tracks the CEO, VP of Engineering, disgruntled Junior Devs, etc. Includes hidden "personality" traits for each.
- **`system_architecture.yaml`**: The current tech stack (e.g., `database: PostgreSQL`, `frontend: React`, `auth: OAuth2`).

### 2. The Issue Queue (The Narrative Engine)
The simulation is driven by a mock Issue Tracker. 
- A "Director" script generates high-level problems: `[INCIDENT] Database latency spiking`, or `[EPIC] Pivot to Blockchain`.
- The system selects an issue and assigns it to the LLM.

### 3. Multi-Agent Artifact Generation
Resolving an issue requires generating multiple synchronized artifacts. We use a sequential agent chain:
1. **The Planner:** Reads the Issue and the current Architecture. Decides *how* the company reacts. Outputs a plan: "We will migrate to MongoDB. The CTO will send a cheerful email, the devs will complain in Slack."
2. **The Writer (Comms):** Generates the emails and Slack messages based on the Planner's outline and the Organization Chart personas.
3. **The Developer (Tech):** Generates mock Git commit messages, Pull Request descriptions, and post-mortems.
4. **The Architect (Mutation):** Crucially, this agent outputs a JSON patch to update the `system_architecture.yaml`. If they migrated to Mongo, the state must reflect this for future runs.

### 4. Graph-Based Static Generation
The generated artifacts are non-linear. They are stored as Markdown/JSON files with explicit links (e.g., the PR description links to the Slack thread URL).
The static site generator (like Hugo or an Eleventy script) builds these into a wiki-like intranet site. The reader can start at a server outage and click through to find the careless commit that caused it.
