# Project 4: The Daily Procedural (Living Newspaper) - Expanded

## Concept
An algorithmic local newspaper for a simulated fictional town. Each run generates a daily edition containing a front-page story, local politics, weird classifieds, and a weather report. Over time, recurring characters (the mayor, local business owners, eccentric townspeople) develop lives and interact.

## How it's deeper than the Museum Task
It requires simulating multiple independent actors and ongoing background events. A pothole reported on Monday might become a sinkhole by Thursday, and a major political scandal by next month. The output is episodic but requires rigid serial continuity.

## Context Management Strategy: "Story Arcs" and "Character Cards"
We avoid feeding previous newspaper text back into the system. Instead we use discrete state tracking:

1. **Active Story Arcs:** A JSON file maintains a list of 5-10 "Active Arcs". An arc has a title, characters involved, current tension level (1-10), and a 2-sentence summary of the latest development.
2. **Selection & Advancement:** When generating a daily paper, the main script randomly selects 2-3 Active Arcs to progress. It feeds only those specific arc summaries to the agent, instructing it to write articles advancing those plots.
3. **Character Roster:** A database of active townspeople. If an agent wants to quote someone, it queries the roster for an appropriate persona. Output extraction scripts update Character states (e.g., changing their job or status) based on the generated article.

### Critical Vulnerabilities & Mitigations (From Review)

**Procedural Flatness & Endless Arcs:**
*Vulnerability:* Managing the pacing of "Story Arcs" algorithmically is very difficult. If the system relies on random selection to advance tension levels, the output will feel procedurally flat (like radiant quests). Arcs might drag on aimlessly.
*Mitigation:* Story Arcs need structured progression triggers. An arc must have a `stage` (e.g., "Introduction", "Escalation", "Climax", "Resolution"). When tension reaches a threshold (e.g., 8+), the system explicitly prompts the LLM to write a *resolving* event. Once resolved, the arc is moved to an "Archived" list and becomes historical lore, making room for a new active arc.

**Character Amnesia:**
*Vulnerability:* If the "Character Roster" just stores static string bios or job titles, characters won't have memory of the events they were involved in unless that memory is dynamically appended. The Mayor might contradict an emotional statement they made two weeks ago.
*Mitigation:* localized context management for characters. Every time a character is heavily featured in a story arc, their DB entry must be mutated to append to a `recent_memories` array (keeping the last 3-5 major events). When the LLM generates a quote or action for them, it receives their static bio *plus* their recent memories to inform their current mood and motivations.

## Next Steps for Implementation
1. **State JSON Design:** Define the exact JSON schema for `active_arcs.json` and `character_roster.json`.
2. **Arc Advancement Logic:** Write the script that selects arcs for today's paper and rules for when an arc naturally increments its tension level versus when it resolves.
3. **Information Extraction:** Build the pipeline that reads the generated newspaper and figures out *how* to update the tension levels and character memories based on what the "reporter" LLM decided to write.
