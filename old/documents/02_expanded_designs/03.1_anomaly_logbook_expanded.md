# 03.1 The Deep-Space / Anomaly Logbook (Expanded Design)

## Core Concept
An ongoing, infinite narrative simulation told entirely through epistolary logs—daily reports from a remote scientific outpost investigating an incomprehensible anomaly (e.g., a spatial rift, an alien megastructure, or a localized physics breakdown). 

## Detailed Mechanics

### 1. The Two-Tiered Memory Architecture
The primary challenge of a perpetual narrative is context window collapse and hallucination of past events. We solve this using a multi-agent memory system.

#### Tier 1: The Lexicon & Story Bible (State)
- A strictly formatted `JSON` or `YAML` file containing the indisputable "Ground Truth".
- **Fields include:** Current Day (e.g., SOL 452), Crew Status (Alive/Dead/Compromised), Base Integrity (0-100%), Anomaly Properties (Known, Theorized, Debunked).
- **The Summarizer Agent:** Every 5 simulated days, a background agent reads the recent logs and proposes a `diff` to the Story Bible. It must extract hard facts and discard prose.

#### Tier 2: The Vector Memory (RAG)
- All generated logs are chunked and embedded into a local vector database (like ChromaDB or FAISS).
- When generating *today's* log, the Generator Agent is first prompted to output a hidden `<scratchpad>` detailing its creative intent (e.g., "I want to follow up on the spectral analysis from last week"). 
- A similarity search is performed against this intent, pulling 2-3 relevant past logs. These are injected into the final prompt as "Archival Reference".

### 2. The Generator Prompt Structure
The agent receives:
- **System Prompt:** Definitively establishing the tone (clinical, paranoid, exhausted, academic).
- **Current State:** The Story Bible.
- **RAG Context:** The results of the vector search.
- **Random Injector (The "Director"):** To prevent narrative stalling, a lightweight script injects a daily "Condition" or "Event" that the agent *must* incorporate (e.g., `[EVENT: Power failure in Sector 4]`, `[CONDITION: The author is experiencing mild auditory hallucinations]`).

### 3. Output Formatting
The output is strictly parsed. It must contain:
1. Meta-data block (Timestamp, Author ID, Security Clearance).
2. The narrative log (HTML/Markdown).
3. `extracted_entities`: A JSON array of any new characters, rooms, or items introduced, to feed back into the Story Bible.

## Aesthetic & UI
The frontend should resemble a retro-futuristic terminal (Phosphor green or amber text on black, monospace fonts, scanlines). The site builds statically, but the feeling is that of discovering an abandoned terminal.
