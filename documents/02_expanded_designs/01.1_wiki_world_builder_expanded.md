# Project 1: Wiki-World Builder (Procedural Encyclopedia) - Expanded

## Concept
An agentic system that builds a comprehensive, interconnected encyclopedia for a fictional world (or a highly specific, deep-niche non-fiction topic) over thousands of iterations. Each run generates a single wiki article (e.g., a city, a historical figure, a battle, a specific technology). 

## How it's deeper than the Museum Task
Instead of random, isolated articles that just happen to share a CSS format, this project builds a highly structured graph of knowledge. Articles must reference each other correctly, maintain chronological consistency, and avoid contradicting previously established lore.

## Context Management Strategy: The "Knowledge Graph" State
To solve the context window problem, we do **not** feed the agent previous articles. Instead, we maintain a central, structured database (like SQLite or a simple JSON graph). 

1. **Entities & Relationships:** Every time an agent creates a page, a post-processing script extracts key entities (names, dates, places) and inserts them into the DB as "nodes" and "edges".
2. **Targeted Prompting:** When the agent runs, it is given a specific node to write about. The script queries the DB for the node's immediate neighbors (1 or 2 degrees of separation) and provides a concise bulleted list of established "facts" to ensure consistency.
3. **Empty Link Resolution (Constrained):** The agent is encouraged to invent new names/concepts (red links). These are stored in the DB as "stub" requests, and become the seeds for future agent runs, ensuring the world expands organically based on its own generated lore.

### Critical Vulnerabilities & Mitigations (From Review)

**The Exponential Stub Problem:** 
*Vulnerability:* If an agent generates 5-10 red links per article, the queue of "stubs to write" will grow exponentially faster than the agent can write them. 
*Mitigation:* The prompt must explicitly limit the agent to generating a maximum of 1-2 new entity references per run. The system should prioritize existing "stubs" based on how many other pages link to them, rather than strictly FIFO.

**Entity Resolution Fragmentation:**
*Vulnerability:* Relying on an LLM to accurately extract entities and relationships post-generation is prone to entity resolution errors (e.g., creating duplicate nodes for "King John", "John the First", and "The Mad King"). Over thousands of iterations, the graph becomes fragmented and noisy.
*Mitigation:* Implement a strict "Entity Resolution" pipeline step. Before saving new entities to the DB, a smaller, cheaper LLM pass explicitly compares newly extracted entities against a vector search of the existing DB. It must decide to either MERGE with an existing entity or create a NEW entity, preventing proliferation of duplicates.

## Next Steps for Implementation
1. **Define the Database Schema:** `Entities` (id, name, type, summary), `Relationships` (source_id, target_id, relation_type, description).
2. **Build the Extraction Prompt:** A system prompt specifically designed to take a generated markdown article and output a strict JSON list of new edges/nodes.
3. **Develop the "Graph Context" Builder:** A script that takes a target entity ID, traverses the graph to depth 2, and formats those relationships into a bulleted "Lore Rules" prompt.
